Audit eseguito il 2026-05-06 su versione live `8.1.9-A.5.2` (commit `3eab8cf`, chiusura A.5.2). Triggered: pre-promozione Fase A → 8.2.0, verifica robustezza pre-Fase 1 SaaS (multi-tenant, sharing folder, dashboard cliente).

## MEDI / BASSI — overview

L'audit ha prodotto ~25 finding non-critici aggregabili in 4 cluster tematici. Quelli rilevanti per la promozione 8.2.0 erano principalmente debiti di refactor Fase A — chiusi col batch C15 + sub-finding correlati.

**Cluster sicurezza secondaria (main.py)** — 10+ finding di info disclosure, IDOR potenziali, validazione input, log policy. Open. Affrontabili in uno sprint dedicato post-Fase A. Esempi: stack trace Google API leakate in errori 502, password change senza invalidazione JWT esistenti, validazione MIME/magic-number STL solo via `endswith(".stl")`.

**Cluster performance icp_engine** — 12+ finding su KDTree riusabili, Kabsch reimplementato 3 volte, parse_stl duplicate, cluster_comps O(n²). Performance acceptable in produzione su volumi clinici; ottimizzazione opportunistica, non bloccante.

**Cluster code health v3b.html** — listener leak (`addEventListener` 55 vs `removeEventListener` 7), THREE.js dispose mancanti (~16 punti), `setInterval` senza clear, 53 `console.log` lasciati, `setTimeout` temporal hack. Cumulano memory leak su sessioni cliniche lunghe ma non producono crash. Affrontabili a tappe.

**Cluster Fase A residuo** — `pdf_gen.py VERSION` hardcoded e fermo a `7.3.9.093`, score thresholds duplicate (`pdf_gen.py:339-348` vs `icp_engine.score_label()`), palette d3 hardcoded in 9+ punti di v3b.html (`#639922` "Ottimo" come literal). Da chiudere prima della promozione 8.2.0.

## Falsi positivi degli agent

Tre finding sono stati esclusi post-verifica:

- **SOSTITUIRE_TEMPLATE_INFO "ancora hardcoded"** (v3b agent): l'IIFE post-A.5.2 c'era gia', l'agent ha confuso il fallback canonico dentro l'IIFE con definizione attiva.
- **CRLF injection via filename in `Content-Disposition`** (main.py agent): bloccata dallo stack ASGI moderno (Starlette/uvicorn validano valori header). Il vero rischio era XSS via `inline + Content-Type`, distinto e affrontato in C2.
- **`SCANBODY_PROFILES` duplica `registry.SCANBODY`** (icp_engine agent): sono concetti diversi. SCANBODY_PROFILES sono brand di scanner (ScanLogiQ, IPD ProCam, Shining), `registry.SCANBODY` sono tipi di marker (1T3, OS, SR). Confusione di nomi simili.
